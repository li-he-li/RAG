# Conversation Memory

## ADDED Requirements

### Requirement: Conversation Message Storage

The system SHALL persist every user message and assistant response in a `conversation_messages` table in PostgreSQL, keyed by `session_id`.

The table SHALL store: `id`, `session_id`, `role` (system/user/assistant/tool), `content`, `token_count`, `created_at`, `metadata` (JSONB).

#### Scenario: User message persisted after chat request

- **WHEN** a user sends a chat message with `session_id="abc123"`
- **THEN** the system SHALL insert a row with role=`user`, content=<user message>, session_id=`abc123`
- **AND** the token_count SHALL be estimated and stored.

#### Scenario: Assistant response persisted after generation

- **WHEN** the assistant generates a response for session_id=`abc123`
- **THEN** the system SHALL insert a row with role=`assistant`, content=<response>, session_id=`abc123`
- **AND** the token_count SHALL be recorded from the API response.

#### Scenario: Messages ordered by creation time

- **WHEN** messages are loaded for a session
- **THEN** they SHALL be returned in ascending order by `created_at`
- **AND** the order SHALL be deterministic for the same session.

### Requirement: Context Assembly with Token Budget

The system SHALL assemble conversation context by loading historical messages from the same session, constrained by the TokenBudgetManager.

The assembly process SHALL:
1. Load recent messages from `conversation_messages` for the current session
2. Sum token_count values, stopping before exceeding the allocated budget
3. If older messages are truncated, generate a compact summary to replace them

#### Scenario: Full history fits within budget

- **WHEN** a session has 10 messages totaling 2000 tokens
- **AND** the token budget allocates 4000 tokens for conversation history
- **THEN** all 10 messages SHALL be included in the assembled context
- **AND** no summarization SHALL occur.

#### Scenario: History exceeds budget triggers summarization

- **WHEN** a session has 50 messages totaling 10000 tokens
- **AND** the token budget allocates 4000 tokens for conversation history
- **THEN** the system SHALL keep the most recent messages that fit within 4000 tokens
- **AND** the truncated older messages SHALL be replaced with a single summary message
- **AND** the summary SHALL be stored as role=`system`, metadata=`{"type": "compact_summary"}`.

#### Scenario: Summary preserves key facts

- **WHEN** a summarization is triggered
- **THEN** the summary SHALL include: main topics discussed, key entities mentioned, any conclusions reached
- **AND** the summary SHALL be generated by calling the LLM with a summarization prompt.

### Requirement: Multi-Turn Message Assembly for LLM

The `chat.py` service SHALL construct the `messages` array for the DeepSeek API as:
`[system_prompt, compact_summary(如果存在), ...historical_messages, current_user_message]`

This replaces the current 2-message format `[system, user]`.

#### Scenario: First message in session

- **WHEN** a user sends the first message in a new session (no history)
- **THEN** the messages array SHALL be `[system_prompt, user_message]` (same as current behavior)
- **AND** no history loading SHALL occur.

#### Scenario: Third message in session with history

- **WHEN** a user sends the third message in session "abc123"
- **AND** the previous 2 user messages and 2 assistant responses exist in the database
- **THEN** the messages array SHALL be `[system_prompt, user_msg_1, assistant_resp_1, user_msg_2, assistant_resp_2, current_user_msg]`
- **AND** the LLM SHALL have full context of the conversation.

#### Scenario: User references previous context

- **WHEN** a user sends "帮我总结一下上面提到的合同条款风险"
- **AND** the previous messages discussed contract clause risks
- **THEN** the LLM SHALL be able to reference the previous discussion because it is in the messages array
- **AND** the response SHALL correctly summarize the previously mentioned risks.

### Requirement: Casual Chat Direct Memory Path

When the user's message is classified as casual chat (non-legal query), the system SHALL respond using conversation history only, WITHOUT triggering the retrieval pipeline.

The casual chat detection SHALL use the existing `_should_skip_retrieval()` function, extended with additional patterns.

#### Scenario: Greeting uses memory path

- **WHEN** a user sends "你好"
- **THEN** the system SHALL NOT call the retrieval service
- **AND** the system SHALL assemble context from conversation history
- **AND** the system SHALL call DeepSeek with a casual chat system prompt.

#### Scenario: Follow-up question uses memory path

- **WHEN** a user sends "能再详细说说吗？" (no legal keywords)
- **AND** previous messages in the session discussed a legal topic
- **THEN** the system SHALL include previous messages in context
- **AND** the LLM SHALL respond based on the conversation history.

#### Scenario: Legal query still uses retrieval

- **WHEN** a user sends "合同第三条有什么风险？"
- **THEN** the system SHALL use the full retrieval + grounded chat pipeline
- **AND** conversation history SHALL still be included in the context (appended after system prompt).

---

### Requirement: Session Cleanup

The system SHALL provide a cleanup mechanism for expired conversation messages.

#### Scenario: Old sessions cleaned up

- **WHEN** a cleanup job runs
- **THEN** messages for sessions with no activity in the last 24 hours SHALL be deleted
- **AND** the cleanup SHALL be triggered via an API endpoint or scheduled task.

#### Scenario: Active session not cleaned

- **WHEN** a session had activity within the last 24 hours
- **THEN** its messages SHALL NOT be deleted by the cleanup job.
