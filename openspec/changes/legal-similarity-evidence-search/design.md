## Context

该变更面向个人法律从业者的核心检索痛点：在多案件、多文书场景中，需要同时判断“案件整体相似度”和“争议点证据段落相似度”。当前系统缺少可溯源的段落级证据定位能力，导致检索效率低且结果可信度不足。该能力将作为后续 Web 大前端的核心能力之一，而非可插拔插件。

约束：
- 输出必须可溯源到具体文件与行号，支持法律工作中的核验流程。
- 需要兼顾文档级召回与段落级精排，避免仅返回向量分数。
- 方案必须可持续演进，支持后续模型、向量库与前端形态调整。
- 依赖准备必须自动化：首次初始化由系统自动下载/拉取 PostgreSQL、Qdrant、BAAI/bge-m3、bge-reranker-v2-m3，并在运行前完成健康检查。

## Goals / Non-Goals

**Goals:**
- 提供双层相似检索：文档级（案件整体）+ 段落级（争议点证据）。
- 建立统一标识体系：文档大标识（doc_id）与段落小标识（para_id）。
- 所有段落命中返回文件名、行号区间、证据摘录、版本标识。
- 对检索命中提供可解释信息（争议点标签/匹配理由）以增强可信度。
- 定义稳定的检索接口，供后续 Web 大前端直接集成。

**Non-Goals:**
- 本变更不覆盖 OCR/扫描件识别质量优化本身。
- 本变更不定义最终前端视觉交互细节。
- 本变更不引入自动法律结论生成，仅聚焦检索与证据定位。

## Decisions

1. 决策：采用“文档级索引 + 段落级索引”双索引架构。  
   - Rationale: 用户需要先定位相似案件，再核验证据段落；单一粒度无法兼顾召回与精度。  
   - Alternatives considered:
     - 仅文档级索引：召回快但证据定位弱。
     - 仅段落级索引：证据细粒度强，但案件整体相似判断不稳定。

2. 决策：采用层级标识模型（doc_id 作为大标识，para_id 作为小标识）。  
   - Rationale: 满足“整篇文章 + 段落”双层追踪与聚合展示需求。  
   - Alternatives considered:
     - 仅段落唯一标识：聚合到文档维度成本高，前端展示复杂。

3. 决策：检索结果强制携带可溯源元数据（file_name、line_start、line_end、version_id）。  
   - Rationale: 法律场景强调可核验性，必须可定位到原文。  
   - Alternatives considered:
     - 仅返回片段文本：可读但不可审计。

4. 决策：采用“查询理解 -> 向量召回 -> 重排聚合 -> 证据解释”流水线。  
   - Rationale: 在争议点检索中，向量召回需要重排以提高可用性，并输出解释提升信任。  
   - Alternatives considered:
     - 仅向量 Top-K：实现快但命中理由不透明。

5. 决策：对行号稳定性使用“标准化文本基线 + version_id”策略。  
   - Rationale: 原始文档可能更新或格式变化，需保证引用一致性。  
   - Alternatives considered:
     - 直接依赖原始文件行号：更新后易漂移，复核成本高。

6. 决策：采用“自动依赖引导（bootstrap）”机制，系统负责下载并准备数据库与向量模型。  
   - Rationale: 降低环境搭建门槛，避免人工漏装导致的不可用问题。  
   - Alternatives considered:
     - 人工安装依赖：步骤分散、失败点多、跨环境一致性差。

## Risks / Trade-offs

- [Risk] 文本切分粒度不当导致语义丢失或噪声增大 -> Mitigation: 采用“自然段 + 争议点标签”双轨切分并进行离线评估。
- [Risk] 行号在文档版本变化后漂移 -> Mitigation: 返回 version_id，并在重建索引时保留历史映射策略。
- [Risk] 争议点识别不稳定影响检索精度 -> Mitigation: 引入人工标注小样本集进行持续评估与迭代。
- [Risk] 检索链路复杂导致延迟升高 -> Mitigation: 分层缓存（查询向量/候选集）与分阶段性能预算。
- [Risk] 自动下载在弱网或受限网络环境下失败 -> Mitigation: 增加重试、断点续传与离线镜像源配置能力。

## Migration Plan

1. 定义并冻结检索接口契约（含文档级+段落级返回结构）。
2. 实现初始化 bootstrap：自动下载/安装 PostgreSQL、Qdrant 与 BAAI/bge-m3、bge-reranker-v2-m3，并完成健康检查。
3. 完成文档与段落数据模型扩展（标识与溯源字段）。
4. 构建索引与检索流水线（召回、重排、解释）。
5. 以灰度数据集验证质量与性能后接入 Web 大前端主流程。
6. 若回滚，降级为仅文档级检索并保留现有检索入口。
