## Context

当前前端已经具备主聊天窗口、会话管理、流式回答、侧边面板和文件上传基础能力，但“合同审查”仍只是左侧栏中的占位入口。后端已有文档上传、解析、数据库持久化和流式聊天能力，这些能力适合复用到标准模板库管理和合同审查回答链路中，但不适合直接承载待审合同，因为用户已经明确待审合同只用于临时审查，不应进入数据库或向量索引。

这次变更的核心约束有三条：
- 合同审查主要在主聊天窗口中完成，而不是单独页面。
- 标准模板属于持久化资产，由数据库管理。
- 待审合同属于临时输入，只在当前会话有效，不入库。

## Goals / Non-Goals

**Goals:**
- 将合同审查能力并入主聊天输入区，作为会话级模式切换的一部分。
- 支持在合同审查模式下上传一个或多个待审合同，并在当前会话内临时保留。
- 基于数据库中的标准模板对待审合同做差异比对，并通过主聊天窗口流式输出自然语言审查结果。
- 提供模板推荐能力，并允许用户在审查前手动改选模板。
- 将左侧“合同审查”面板转为标准模板库管理面板。

**Non-Goals:**
- 不将待审合同写入数据库、文件管理列表或向量索引。
- 不实现审查历史记录、跨刷新恢复或长期会话缓存。
- 不在第一版接入案例检索、通用规则引擎或复杂结构化报告页面。
- 不改变普通聊天模式的既有能力和 API 契约。

## Decisions

1. 决策：前端用单一聊天输入区承载两种模式：普通聊天与合同审查。  
   - Rationale: 用户明确要求大模型回答尽量留在主聊天窗口中，复用现有聊天与流式展示比新增独立主视图更稳。  
   - Alternatives considered:
     - 继续保留独立合同审查面板作为主入口：交互割裂，无法复用主聊天流。
     - 单独新增合同审查页面：实现成本更高，也偏离当前产品方向。

2. 决策：左侧“合同审查”只负责标准模板库管理，不负责待审合同上传。  
   - Rationale: 标准模板是持久化资产，待审合同是临时会话输入，混用会使数据边界失控。  
   - Alternatives considered:
     - 所有合同都复用文件管理或模板面板上传：会把待审合同误纳入持久化体系。

3. 决策：待审合同通过独立临时上传链路处理，按会话存储解析结果和最少元数据。  
   - Rationale: 满足“不入数据库”的要求，同时允许同一会话内改选模板后重复发起审查。  
   - Alternatives considered:
     - 请求结束即删除：无法支持用户在同一会话中重复试审。
     - 直接持久化到数据库：违背已确认的产品边界。

4. 决策：模板匹配采用“系统先猜、用户可改”的双阶段流程。  
   - Rationale: 纯自动匹配误判风险高，纯手动选择增加操作成本，双阶段更符合第一版落地平衡。  
   - Alternatives considered:
     - 只自动猜：错误模板会直接影响审查结果。
     - 只手动选：对用户来说路径更重。

5. 决策：多文件审查只做串行流式输出，不做并发混流。  
   - Rationale: 现有聊天窗口以线性流为主，串行输出最容易保持可读性和实现稳定性。  
   - Alternatives considered:
     - 并发输出多个文件结果：会打乱主聊天流和用户阅读顺序。
     - 第一版只做单文件：不满足已确认的批量需求。

6. 决策：第一版审查提示词只围绕“标准模板差异”组织回答。  
   - Rationale: 先把模板差异主链路做准，避免引入案例检索、规则引擎和评分体系导致范围失控。  
   - Alternatives considered:
     - 同时叠加规则审查或案例检索：复杂度明显升高，且当前需求并未要求。

## Risks / Trade-offs

- [Risk] 临时文件如果只放在前端状态中，后端流式审查过程可能无法复用已上传内容 -> Mitigation: 使用服务端会话级临时存储，并以临时文件标识回传给前端。
- [Risk] 模板推荐错误会导致审查基准偏差 -> Mitigation: 前端必须展示推荐模板和候选列表，并允许用户改选后再发送。
- [Risk] 多文件串行审查会拉长单次回答时长 -> Mitigation: 通过流式输出逐份返回结果，避免用户长时间无反馈。
- [Risk] 普通聊天与合同审查共用输入区容易造成状态混淆 -> Mitigation: 使用显式高亮模式按钮、模式专属上传入口和发送前校验。
- [Risk] 标准模板与普通文件管理边界不清会导致后续接口混乱 -> Mitigation: 在 API 与前端面板职责上明确区分模板库与临时待审合同链路。

## Migration Plan

1. 新增 OpenSpec 规格，冻结合同审查模式、模板库管理和临时文件边界。
2. 调整前端聊天输入区交互，加入模式切换与待审合同上传入口。
3. 将左侧“合同审查”面板从占位改造为标准模板库管理界面。
4. 新增后端临时上传与合同审查流式接口，并接入模板推荐能力。
5. 验证普通聊天模式与合同审查模式可以并存，且待审合同不会出现在持久化文件列表中。
6. 如需回滚，可保留模板库面板并禁用合同审查模式入口，普通聊天链路不受影响。
