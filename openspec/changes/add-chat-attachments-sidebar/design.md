## Context

当前系统已经有两条文件相关链路：一条是文件管理中的持久化知识库文档，另一条是合同审查模式下的临时待审合同。普通聊天模式虽然已经共享同一个输入区，但还缺少“上传文件给 AI 直接阅读”的轻量附件链路。与此同时，右侧栏现在只能显示引用来源，不足以承载会话附件列表。

这次变更的核心约束有四条：
- 普通聊天附件必须和知识库文档、合同审查待审合同、标准模板库分开管理。
- 普通聊天附件必须是会话级的，而不是长期持久化资产。
- 右侧栏需要保留现有引用来源能力，并通过顶部标签页承载附件视图。
- 同一个左下角上传按钮需要根据当前模式路由到不同的上传链路。

## Goals / Non-Goals

**Goals:**
- 支持在普通聊天模式下上传一个或多个会话级附件。
- 让主聊天回答可以读取当前会话附件内容。
- 在右侧栏顶部加入标签页，并支持在“附件”和“引用来源”之间切换。
- 在右侧栏附件标签页中显示当前会话已上传的普通聊天附件。
- 保持普通聊天附件与合同审查待审合同、标准模板库、知识库文档之间的数据边界清晰。

**Non-Goals:**
- 不把普通聊天附件写入数据库、向量索引或文件管理列表。
- 不实现跨会话、跨刷新长期恢复的附件缓存。
- 不把右侧栏改造成独立页面或新增新的主内容区。
- 不改变现有引用来源的基本展示语义，只改变其容器形态与切换方式。

## Decisions

1. 决策：普通聊天附件采用会话级临时存储，而不是复用现有文件管理持久化链路。  
   - Rationale: 用户要的是“上传给 AI 读”的即时能力，而不是把附件沉淀成知识库资产。  
   - Alternatives considered:
     - 直接复用文件管理上传：会把普通聊天附件误写入持久化文档体系。
     - 仅把文件留在前端内存：后端聊天请求无法稳定复用附件内容。

2. 决策：继续复用同一个左下角上传按钮，并按当前模式切换上传目标。  
   - Rationale: 交互入口保持稳定，用户不需要记两个按钮；普通聊天和合同审查只在上传目标上分流。  
   - Alternatives considered:
     - 给普通聊天和合同审查各加一个上传按钮：输入区会变得拥挤，且视觉层级混乱。
     - 只在普通聊天开放上传：会削弱合同审查模式的一致性。

3. 决策：右侧栏改为单容器多标签页结构，顶部至少提供“附件”和“引用来源”两个标签。  
   - Rationale: 可以保留现有右侧栏位置与开合行为，同时承载新增附件列表。  
   - Alternatives considered:
     - 新增第二个侧栏：布局复杂度高，移动端适配更差。
     - 用引用来源覆盖附件列表：会让两种信息互相抢占，用户难以回看。

4. 决策：普通聊天请求显式携带当前会话附件引用，并由后端在回答前组装附件上下文。  
   - Rationale: 这样附件是否可读成为明确契约，便于后续做大小限制、失败处理和调试。  
   - Alternatives considered:
     - 只在前端展示附件，不进入回答上下文：满足不了“AI 可读”。
     - 隐式从服务端读取全部临时文件：上下文来源不透明，也不利于会话隔离。

## Risks / Trade-offs

- [Risk] 普通聊天附件过大可能显著拉长回答延迟 -> Mitigation: 限制类型和大小，并在右侧栏或上传反馈中提示处理状态。
- [Risk] 右侧栏标签页增加后，用户可能不清楚自己当前看到的是附件还是引用来源 -> Mitigation: 顶部标签页高亮当前选项，并在特定动作下自动切到对应标签。
- [Risk] 三类文件链路并存后，普通聊天附件、合同待审文件和知识库文档容易混淆 -> Mitigation: 在命名、接口、前端列表和清理策略上强制分离。
- [Risk] 同一个上传按钮模式切换路由错误会把文件送到错误链路 -> Mitigation: 上传前根据当前模式做显式分支，并在服务端再次校验类型和目标。

## Migration Plan

1. 增加普通聊天附件与右侧栏标签页的 OpenSpec 规格与任务。
2. 扩展前端会话模型，加入普通聊天附件状态与右侧栏标签状态。
3. 调整输入区上传按钮逻辑，使普通聊天模式与合同审查模式分流到不同上传链路。
4. 新增后端普通聊天附件临时上传与上下文读取能力。
5. 将右侧栏从单一引用面板升级为标签页面板，并接入附件列表。
6. 验证普通聊天附件、引用来源、合同审查待审合同三条链路不会互相污染。
