## Context

当前项目已经具备三类与本次变更直接相关的基础能力：一是类案检索与证据溯源，二是普通聊天与合同审查的模式切换，三是会话级与持久化文件的分层管理。其中，普通聊天和合同审查已经证明“左侧管理页 + 聊天区模式开关 + 主聊天流渲染结果”的交互框架可行，但系统仍缺少一个专门面向法律攻防分析的“对方观点预测”链路。

本次变更的关键约束已经明确：
- 左侧“观点预测”页只做模板后台管理，不在左侧页直接发起预测。
- 案件模板由案件名称、案情材料和可选对方语料组成，支持多个案件并存。
- 用户在聊天区发起预测时，必须同时提供自然语言问题和所选模板。
- 模板材料允许持久化入库，但不得混入现有主检索文档库。
- 删除直接采用硬删除，不能保留软删除过渡层。

相关利益相关方包括：法律从业者用户、后续基于 OpenSpec 拆分实现任务的开发者，以及依赖现有聊天/合同审查能力不被破坏的现有流程。

## Goals / Non-Goals

**Goals:**
- 提供独立的“观点预测”模式，并保持与合同审查一致的交互心智。
- 提供可复用的案件模板后台，支持多案件并存、选择和删除。
- 将预测输入定义为“用户自然语言 + 所选模板”的联合输入。
- 输出结构化的预测报告卡片，包含对方观点、依据、反制抓手和信息缺口。
- 为观点预测建立独立的数据模型和后端链路，避免污染现有文档检索库。

**Non-Goals:**
- 不实现裁判结果预测、胜诉率评分、法官画像或地域倾向分析。
- 不将观点预测模板材料纳入普通文档管理、模板库或主检索索引。
- 不在第一版中实现复杂的模板编辑、版本管理、协作共享或回收站机制。
- 不引入开放式多智能体社会仿真，仅保留固定法律分析视角。

## Decisions

1. 决策：左侧“观点预测”页仅作为模板后台，不承担预测执行入口。  
   - Rationale: 用户已经明确区分“模板管理”和“使用模板发起分析”两个动作；将发起动作放在聊天区可复用现有聊天模式心智，并避免左侧页承担双重职责。  
   - Alternatives considered:
     - 在左侧页直接点击“开始预测”：会让模板后台和对话主链路耦合，削弱主聊天区作为统一工作台的定位。
     - 让左侧页同时显示预测结果：会和现有主聊天流冲突，也不利于后续统一历史会话体验。

2. 决策：案件模板由 `case_name + case_material + optional opponent_corpus` 组成，保存后在聊天区单选使用。  
   - Rationale: 该结构与用户对“多个案子并存”的需求一致，且能保证观点预测总是围绕一个明确案件上下文执行。  
   - Alternatives considered:
     - 把模板拆成多个松散材料由用户每次临时拼装：操作成本高，也会导致当前任务上下文不稳定。
     - 允许一次选择多个模板：会显著增加画像抽取和结果归因复杂度，第一版不利于控边界。

3. 决策：观点预测请求必须同时携带用户自然语言和 `template_id`。  
   - Rationale: 模板只提供背景，不代表用户本次想问什么；自然语言只提供意图，不足以承载案件事实。联合输入是该工作流的核心约束。  
   - Alternatives considered:
     - 仅基于模板自动预测：结果泛化过强，缺少任务目标。
     - 仅基于自然语言：无法稳定绑定具体案件，尤其不适合多案件并存场景。

4. 决策：观点预测材料进入独立数据库表，不复用现有 `documents / paragraphs` 主库。  
   - Rationale: 现有主库承载类案检索和合同模板等持久化资产，如果将案件模板材料混入，会污染主检索范围并增加删除/隔离风险。  
   - Alternatives considered:
     - 继续复用 `documents` 并靠 `source_path` 前缀区分：短期可行，但会让主检索库语义越来越混乱。

5. 决策：预测链路采用“案件画像抽取 -> 对方立场检索 -> 观点树生成 -> 我方反制层 -> 报告卡片”的中间层架构。  
   - Rationale: 法律场景需要可复核的中间结构，不能把“对方观点预测”退化为一次性聊天生成。该设计也吸收了 MiroFish 中“先建中间结构，再出结果”的优点。  
   - Alternatives considered:
     - 直接把模板内容拼进 LLM prompt 生成答案：实现快，但输出不稳定、难以解释，也难以沉淀结构化报告。

6. 决策：删除策略直接采用硬删除，并同步清理模板、材料、切段结果和关联快照。  
   - Rationale: 用户已经明确拒绝软删除，同时模板页定位更偏后台管理，硬删除的心理预期更一致。  
   - Alternatives considered:
     - 软删除 + 延迟清理：安全性更高，但与用户当前偏好不符，也会引入额外运维复杂度。

7. 决策：结果在主聊天区以“预测报告卡片”呈现，而不是普通消息正文。  
   - Rationale: 法律分析结果需要固定结构，报告卡片更适合展示案件摘要、观点节点、引用依据和反制建议，也便于后续复用为导出或归档对象。  
   - Alternatives considered:
     - 普通聊天气泡：阅读成本高，结构不稳定，不利于做引用和模块化展示。

## Risks / Trade-offs

- [Risk] 模板材料进入独立库后，和主检索库之间存在重复解析逻辑 -> Mitigation: 复用现有解析与切段服务，但保持存储边界分离。
- [Risk] 硬删除模板时，用户可能误删仍在使用的案件背景 -> Mitigation: 前端删除前做明确确认，并在删除当前已选模板时立即清空选择器和阻止发送。
- [Risk] 对方语料选填会导致部分案件预测依据不足 -> Mitigation: 报告中显式输出信息缺口，并允许系统在无对方语料时只基于案情材料和检索结果做保守预测。
- [Risk] 多案件并存会导致用户错选模板 -> Mitigation: 聊天区模板选择器固定显示案件名称，并在报告卡片顶部重复展示当前使用模板。
- [Risk] 结构化观点树增加链路复杂度和延迟 -> Mitigation: 使用阶段状态流逐步反馈，避免长时间无响应。

## Migration Plan

1. 新增 OpenSpec capability 与对应 specs，冻结模板管理和观点预测模式的行为契约。
2. 新增观点预测域数据库表和后端 schema，完成模板、材料与结果对象建模。
3. 扩展左侧菜单和聊天区 composer，加入模板后台、模板选择器和观点预测模式开关。
4. 实现模板创建、列表、删除、材料上传与详情接口。
5. 实现观点预测后端链路和任务状态接口，打通案件画像、检索、观点树、反制层和报告卡片。
6. 完成回归验证，确保普通聊天和合同审查不受影响。
7. 回滚方案：隐藏观点预测入口并停用新接口；独立数据表保留但不参与现有主流程。
